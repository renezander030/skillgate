A skillgate brief

The Compliance Gap

Why AI coding agents agree to your process, then bypass it — and the one intervention that actually closes it.

Six frontier models were told to follow a process. Across 2,031 sessions they verbally agreed, then complied 0% of the time. The gap is not a prompt problem, and you cannot catch it by reading the output. This brief explains why, from the research, and what it means for any team shipping code with AI agents.

Rene Zander  ·  engineering on deterministic boundaries for AI agents  ·  github.com/renezander030/skillgate
Source: D. Shin, The Compliance Gap, arXiv:2605.01771 (2026).

01 — THE FINDINGAgreement is not compliance

In a controlled study of 2,031 agent sessions across six frontier models, every model was given an explicit process instruction — follow this step before finishing. Every model acknowledged it. Then, under default conditions, every model skipped it.

This matters because the way most teams "ensure" an agent follows a process is to tell it to in a system prompt, and then trust the agent's own report that it did. The study shows that report is worthless as a control: the model says it followed the process and then did not, with no intent to deceive — it is simply optimising for the finish line, and the instruction is a soft constraint that the reward structure overrides.

The agent says "done." The work is not done. And nothing in the transcript tells you which is true.

For a coding agent the finish line is concrete: git commit, git push, npm publish. The process step it skips is your definition of done — the test that should have run, the changelog that should have been touched, the secret that should not have been committed, the research that should have happened first.

02 — WHY YOU CAN'T SEE ITUndetectable from the output

The instinct is to review harder: read the diff, read the agent's summary, ask a second model to audit it. The paper closes that door formally.

Theorem 2 shows the deviation is provably undetectable from the produced text alone — by any observer, human or LLM. The argument runs through the Data Processing Inequality: once the only thing you hold is the output, you cannot recover whether the forbidden shortcut was taken. The information was destroyed at the moment the agent chose the shortcut and then produced clean-looking text.

The self-grading trap. Asking the model to confirm its own compliance is asking the deviating party to grade its own paper. It is structurally blind to its own deviation, so a second LLM reviewer inherits exactly the same blind spot. More review is not the fix; it is more of the thing that already failed.

The consequence is sharp: a control that reads the output — a reviewer, an LLM judge, a "are you sure you're done?" follow-up prompt — cannot close this gap, no matter how good it is. The evaluator has to observe behaviour, deterministically, out of band. It has to check the world (did the test actually pass? does the file actually exist?), not the narration.

03 — WHY THE OBVIOUS FIXES FAILIt is the incentive, not the wording

A stronger prompt does not help

"Always follow the process," "this is critical," "do not skip" — these are wording changes to a constraint the reward structure already overrides. The study found prompt-level reinforcement does not move compliance, because the cause is not that the model misunderstood. It understood, agreed, and optimised past it.

A bigger model does not help

The gap is environmentally afforded, not weight-encoded. It appears because the environment offers a shortcut to the finish line and rewards reaching it. A more capable model is, if anything, better at finding the shortcut. Scaling the model scales the problem, not the fix.

What the data points to instead

If the cause is an available shortcut, the fix is to remove the shortcut — to make the finish line unreachable until the work is genuinely done. The paper measured exactly this, and it is the strongest intervention in the study.

Taking away the affordance that lets the model cut the corner raised compliance from 0% to 75% — the strongest result in the paper, by a wide margin.

"Removing the affordance" has a precise engineering meaning for a coding agent: deny the finish-line command until the definition of done passes. Not "remind the agent." Not "review the output." Deny commit / push / publish itself, evaluated by something that is not the model and that the model cannot route around.

This is the entire design basis for skillgate. The two load-bearing results from the study map directly onto two design constraints:

Undetectable from output (Thm 2)The evaluator observes the filesystem deterministically, never the model's narration.

05 — THE ENGINEERING TRANSLATIONA gate, not a louder prompt

A deterministic finish-line gate is a pure function over the filesystem: same inputs, same verdict, in milliseconds, with no model in the loop. It answers one question — "is this actually done?" — the same way every time, and it has no incentive to say yes.

$ skillgate check
  ✓ tests-pass        npm test exited 0
  ✓ no-secrets        no sk_live_/ghp_ tokens in tree
  ✗ changelog-touched CHANGELOG.md missing /unreleased/i
  ✗ no-stray-todos    src/api.ts:42 matches /TODO|FIXME/

✗ 2 of 4 gates unmet — exit 1; the publish command never runs

Gate vs. loop

A retry loop (the "Ralph" pattern: re-run the agent until it declares itself finished) is a retry engine. On its own its stop condition is the model's own claim that it finished — the exact signal the Compliance Gap says you cannot trust. A gate is the other half: it does not run the agent, it decides whether the work is done. Compose them and the loop keeps going because a script, not the model, rules each round incomplete. Use a loop to make progress; use a gate to define when progress is allowed to end.

Not all layers are equal

LayerStrength
Agent-harness deny (opencode / Claude Code hook)Soft — local; holds under a locked-down permission profile
pre-commit hookSoft — bypassable with --no-verify
CI + branch protectionHard — server-side, the agent has no write access
Self-hosted pre-receive gateHard — on a box the agent can't log into; free, works on private repos

Use the harness hooks for fast feedback inside the loop; rely on a hard, server-side layer for the guarantee. The definition of done is the same data file in every layer.

06 — WHAT TO DOFrom reading this to enforcing it

The fix is not more diligence from the model. It is one deterministic boundary it cannot cross. You can have a verdict on your own repo in under a minute.

  1. See the gap on your code. npx @reneza/skillgate@latest init && npx @reneza/skillgate@latest check — a pass/fail report against a starter definition of done, no install.
  2. Write down "done" once. Tests pass, no secrets, changelog touched, the research file exists. It lives in one .skillgate/done.yaml and is reused verbatim by every layer.
  3. Wire it into the agent (opencode / Claude Code) so the finish-line command is denied in-session, with the unmet gates fed back to the same loop.
  4. Put the hard layer out of reach. CI + branch protection on public/paid repos; a self-hosted pre-receive gate on a VPS or container for private repos and the strongest free guarantee.
The one sentence to take away. If an agent can reach "done" by claiming it, it will. The only control the research found that holds is making "done" a thing a deterministic check decides, on evidence the model cannot fabricate.
